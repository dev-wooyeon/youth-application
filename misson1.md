# 1. 학습 과정과 현재 자신이 생각하는 역량은 무엇인가요? (1000자 이내)

"저는 야생형으로 배우는 사람입니다. 모르면 당장 해봐야 직성이 풀리고, 알게 되면 기반이 되는 이론을 학습해나갑니다. 기초는 책으로 다지고, 실무에서 바로 써야 하는 건 온라인 강의로 빠르게 해결합니다. 배운 건 꼭 제 언어로 정리하고, 제 생각을 덧붙여 기록으로 남깁니다. 나중에 다시 꺼내 봐도 ‘왜 이렇게 생각했는지’가 보이도록 노력합니다.
(https://github.com/dev-wooyeon/reading-books-for-programmers)

제가 가진 역랑은 ‘단순화’와 ‘자동화’입니다. 복잡한 걸 단순하게 만드는 건 기술보다 생각의 정리라고 믿습니다. 읽기 쉬운 코드를 고민하고, 반복되는 일을 자동화해 팀의 리소스를 아끼는 걸 좋아합니다. 예전에 정산 배치 업무를 자동화해서 운영 담당자분들의 퇴근 시간을 앞당기며 긍정적인 피드백을 받은 경험으로, 개발하는 일을 즐기고 있습니다.

직무적으로는 PG 정산팀 3년, 테마파크 IoT팀 2년을 거치며 다양한 도메인을 경험했습니다. Java와 Spring Boot를 주력으로 쓰지만, 문제 해결하기에 필요한 것이라면 특정 기술에 매몰되지 않습니다. 실제 사례로 Python기반의 어플리케이션 유지보수까지 맡기도하고 간단한 NFC Reader 프로그램도 만들어 본 경험이 있습니다.

리더십은 지급대행 프로젝트를 하면서 처음 배웠습니다. 3명 규모의 소규모 프로젝트 이었지만 기술 스택을 전환하고 일정과 리소스를 조율하며 팀을 이끈 경험이 있습니다. 힘들었지만, 덕분에 협업의 중요성과 ‘정답을 찾아가는 과정’을 배웠습니다.
개인적으로는 실행력을 가장 큰 자산으로 생각합니다. 배우고 싶으면 스터디를 직접 만들고, 꾸준히 운영해왔습니다. 멘토링도 그 연장선입니다. 지금까지 100명 넘는 주니어 개발자들과 함께했고, 그 과정에서 제가 더 많이 배우곤 했습니다.

결국 제 학습과 역량의 중심엔 실행이 있습니다. 모르면 배우고, 알면 개선하고, 익숙해지면 자동화합니다. 복잡한 걸 단순하게, 불편한 걸 효율적으로 바꾸는 일을 가장 잘하는 것 같습니다."

# 2. 선택한 직군을 목표로 하게 된 이유는 무엇인가요? (1000자 이내)

제가 지금의 직군을 목표로하게 된 이유는, 단순히 ‘개발로 사람들의 편리함을 가져다 주는 것’에서 시작했지만, 결국 ‘데이터를 잘 다루는 개발자’로 성장하고 싶다는 목표로 이어졌기 때문입니다.
정산팀에서 3년간 일하며 매일 수만 건의 거래 데이터를 다뤘습니다. 처음엔 단순히 정합성을 맞추는 일이었지만, 어느 순간부터 ‘이 데이터를 더 좋은 방향으로 만들순 없었을까?’라는 고민으로부터 시작했습니다. 
배치 자동화를 구축하고, 오류 케이스를 분류하며 시스템 안정성을 높이면서 깨달은 건 하나였습니다. 백엔드 로직보다 더 본질적인 건 데이터의 구조와 흐름이라는 점이었습니다.
이후 테마파크 IoT 팀으로 이직하며 센서, 하드웨어 등의 데이터를 실시간으로 수집되고, 가공되어, 최종적으로 분석 테이블로 쌓이는 과정을 경험했습니다.

서버 개발자 입장에서 데이터를 저장하고 꺼내는 수준을 넘어, ‘데이터의 생애주기’를 설계하는 관점을 배우게 되었습니다. 이 경험은 자연스럽게 저를 데이터 엔지니어로 이끌었습니다.
저는 여전히 백엔드 엔지니어이지만, 제 시선은 데이터 엔지니어의 일에 닿아 있습니다. Spark, DuckDB, Airflow 같은 툴을 공부하며 파이프라인을 직접 설계해보고 있습니다. 시스템을 단순히 안정적으로 “운영하는 사람”이 아니라, 데이터를 통해 “의미를 추출하고 개선 방향을 제시하는 사람”이 되고 싶습니다.
저는 시스템 구조와 데이터 흐름을 설계하는 데 강점이 있습니다. 
업무에서 반복되는 패턴을 자동화하고, 사람의 개입을 줄이는 구조를 만드는 걸 즐깁니다. 그리고 그 과정에서 효율성뿐 아니라 ‘이 데이터가 나중에 어떻게 쓰일 수 있을까’를 항상 고려합니다.

결국 제가 이 직군을 목표로 하는 이유는 명확합니다. 데이터 엔지니어는 제가 그동안 쌓아온 정확성과 효율성 중심의 백엔드 경험을 가장 잘 확장할 수 있는 영역이기 때문입니다. 시스템의 안정성을 다루던 사람이 이제는 데이터의 가치를 다루는 사람으로 성장하고자 합니다.

# 3. 마음이 끌리는 무언가에 몰입하여 일정 수준의 결과물을 만들어 낸 경험이 있나요? (1000자 이내)
저는 새로운 기술을 배우면 꼭 써봐야 직성이 풀립니다. 머리로만 이해하면 남의 지식이지만, 직접 만들어보면 제 것이 된다고 믿습니다.

최근에는 ‘바이브 코딩(Vibe Coding)’을 익히기 위해 Codex와 AWS Kiro를 활용해 약 1주간 개인 프로젝트를 진행했습니다. 
주제는 ‘게임 공지사항 알림 서비스’였습니다. 좋아하는 게임의 업데이트 소식을 자동으로 감지해 사용자에게 알려주는 기능을 만들고 싶었습니다. 
단순히 알림 기능이 아니라, 게임 커뮤니티와 유저 간의 정보 공백을 줄이는 서비스를 목표로 잡았습니다.

혼자서 기획부터 개발, 배포까지 진행했습니다. 백엔드 로직은 Codex 기반으로 자동 코드 생성을 실험하며 설계했고, 프론트는 Kiro를 활용하여 구현했습니다. 
프로젝트 배포는 프론트는 Vercel, 백엔드는 fly.io를 활용하였습니다.
이 프로젝트는 주기적 스크랩을 통해 트리거를 통해 게임사 공지 페이지의 변동 사항을 감지하고 메일로 보내는 방식이였습니다.

이 과정에서 저는 AI 도구를 단순히 사용하는 법이 아니라, 협업하듯 활용하는 방법을 배웠습니다. 
Codex가 생성한 코드를 그대로 쓰지 않고, 왜 그 로직이 나왔는지를 분석하고 수정했습니다. 생성형 AI의 한계와 가능성을 동시에 경험하면서, 기술을 ‘도구’로써 다루는 감각이 생겼습니다.

서비스는 실제로 배포해 사용했습니다. 게임을 좋아하는 친구들에게 베타 테스트를 진행했고,
“일반적으로 메일보다는 카톡이나 디스코드를 많이 사용하기 때문에 알림 채널을 변경하면 좋을 것 같다” 등의 많은 피드백을 받았습니다. 
하지만 대규모의 게임의 공지사항은 사전에 방송을 통해 먼저 고객들에게 알려주기 때문에 새로운 공지사항은 느리다는 판단을 했고, 서비스를 중단했습니다.
하지만 이 경험 덕분에 좋은 제품을 만들기 위해서는 사전 시장 조사가 중요하다는 걸 명확히 배웠습니다.

돌이켜보면, 이 프로젝트는 제게 단순한 사이드 작업이 아니라 하나의 몰입 경험이었습니다.
아이디어에서 배포까지 모든 단계를 혼자 감당하며, 문제를 해결하는 과정 자체가 즐거웠습니다.
결과적으로 AI를 이용한 서비스 설계 경험, 혼자서 서비스를 완성할 수 있는 실행력, 그리고 한계를 인식하고 멈출 줄 아는 객관성까지 얻었습니다.

저는 이런 과정을 경험하면서 무언가에 몰입한다는 건 결국 “원하는 걸 얻을 때 까지 지치치않고 실행할 수 있는 힘”이라고 생각합니다.

# 4. 우리 회사에 지원하려는 이유는 무엇인가요? (500자 이내)
과거 정산팀에서 3년 동안 월 100만건 수준의 거래 데이터를 다루며 시스템 안정성과 효율을 고민해왔습니다. 
반복 업무를 자동화하고, 데이터 흐름을 개선하면서 ‘운영 효율’을 높이는 경험을 쌓았습니다. 
이 경험은 ㅇㅇ사처럼 대규모 데이터를 다루는 조직에서 즉시 기여할 수 있는 부분이라고 생각합니다.

반면, 데이터 엔지니어링 영역은 아직 깊게 다루지 못했습니다.
그래서 체계적인 데이터 인프라 환경에서 데이터를 설계하고 다루는 경험을 쌓고 싶습니다. 
백엔드에서 다져온 구조적 사고와 자동화 역량을 바탕으로, 데이터를 통해 더 큰 임팩트를 내는 개발자로 성장하고 싶습니다.
